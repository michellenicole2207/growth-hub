# Chapter 1 - Fundamentals of Testing

## Key Concepts
- Testing is not only about finding defects; it also builds confidence in quality
- Testing and debugging are different activities
- Testing contributes to quality by preventing defects (shift-left)
- Testing is context-dependent — no single approach works for everything
- Exhaustive testing is impossible; risk and priorities guide testing effort

## The 7 Testing Principles
1. Testing shows presence of defects, not their absence
2. Exhaustive testing is impossible
3. Early testing saves time and money (shift-left)
4. Defects cluster together (Pareto principle)
5. Tests wear out — vary and update them (pesticide paradox)
6. Testing is context dependent
7. Absence of defects is a fallacy

## Key Terms
- **Defect**: A flaw in a component or system
- **Error**: A human mistake that produces a defect
- **Failure**: When a defect is executed and causes incorrect behavior
- **Root cause**: The fundamental reason a defect exists
- **Test basis**: Documentation used to design test cases
- **Test object**: The item being tested
- **Testware**: All artifacts produced during testing
- **Verification**: Are we building the product right?
- **Validation**: Are we building the right product?

## Test Activities & Tasks
- Test planning
- Test monitoring and control
- Test analysis (what to test)
- Test design (how to test)
- Test implementation (preparing testware)
- Test execution
- Test completion

## Testing in Context
- **Static testing**: Reviewing without executing code (reviews, walkthroughs)
- **Dynamic testing**: Executing the software
- **Shift-left**: Testing earlier in the lifecycle
- **Whole team approach**: Everyone is responsible for quality
